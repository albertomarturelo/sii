# Wire contract — F29 (Declaración Mensual de IVA)

First-hand observation, no third-party library (ADR-004). All values below are
**synthetic / redacted** — folios, RUTs, montos, names are PII/taxpayer data and MUST
NOT land here. Scope is **read-only**. Built in two phases (ADR-013).

- **Fase 1 (shipped, robust):** the SDI-**JSON** facades — propuesta + estado.
  **Live-validated from the TS port 2026-06-29** (prod, empresa session): `f29
  formulario`/`overview`/`status` returned real data.
- **Fase 2 (deferred):** the **presented form** via `rfiInternet` **GWT-RPC** — the
  full filed código grid. Mapped in the spike (below); built later (own PR + ADR).

Surface: `@altumstack/sii-core` `portal/f29.ts` (`fetchF29Propuesta`, `fetchF29Estado`) +
`portal/f29-codigos.ts` (the código taxonomy), driven by the `f29Formulario` /
`f29Overview` / `f29Status` tasks under `withSession`.

**SESSION-KEYED (ADR-005).** F29 authorizes by the **session principal**, confirmed
live (Python 2026-06-26): a represented empresa's body RUT → `Consulta RUT no esta
autorizado`. The tasks take NO `--rut` and **reject a representing operate pointer up
front** (before any session) with an actionable "log in as the empresa" message.

## Fase 1 — SDI JSON facades (`propuestaf29ui`)

Base: `https://www4.sii.cl/propuestaf29ui/services/data/facadeAdapterService/`
(host from config `HOSTS.portalApi`). Envelope: the shared SDI shape `{ metaData,
data }`; error channel is **`metaData.errors`** (a list of `{id, descripcion}`) — like
F22, NOT RCV's `respEstado`. `conversationId` = the session `TOKEN` cookie;
`transactionId` = a per-call UUID. Headers: `Referer` = the propuestaf29ui SPA root.

| Facade | Purpose | `data` shape |
| --- | --- | --- |
| `getDeclaracionConCondicionesYTipoPropuesta` | IVA **propuesta** (SII's pre-fill) | object (or `null` = sin propuesta) |
| `getDeclaracionConEstados` | **presented/saved** records (estado) | array (`[]` = nada presentado) |

Namespaces: `cl.sii.sdi.lob.iva.propuestaf29.data.api.interfaces.FacadeAdapterService/<op>`.

**Request bodies** — período split into `mes` (`MM`) + `anno` (`YYYY`); internal form
id `"2"`. SII names the same fields differently per facade (its own quirk, observed):
- propuesta: `{ rutContribuyente, dv, formCodigo:"2", mes, anno }`
- estado: `{ rut, dv, formId:"2", mes, anno }`

**Propuesta response** (`data`, synthetic): `{ tipopropuesta, estado, descripcionEstado,
listCodPropuestos:[{codigo,valor}], listCodAdministrativos:[{codigo,valor}],
listCodBase:[…PII…], resultadoCalculoPP29:{traza:"…RUT…"} }`. `valor` is plain-numeric
(`.`-decimal, no thousands sep) → `Number()`. **PII dropped:** `listCodBase`
(nombre/RUT/dirección) + `traza` (embeds RUT) are never read; NO `raw`. The códigos are
SII's *proposed subset*, served WITHOUT glosas (labeled from the taxonomy below).

**Estado response** (`data[]`, synthetic): `[{ estadoDeclaracionId, estado, folio,
declFechaCreacion, monto, enNegocio, codigo }]`. The `monto` (declared **total a
pagar**) IS surfaced as `total` — the user's own monthly figure, the point of the
overview; it never reaches the audit log. Works for **historical** months. Observed
estados: `1` Vigente, `10` Guardada, `70` "Rechazada por pago inconcluso".

## Código taxonomy (`portal/f29-codigos.ts`)

Observed first-hand from the rendered F29 form HTML (`rfiInternet/cargarHtml`, captured
2026-06-29). The form table gives, per line: **glosa** (`celda-glosa`), **código**
(`celda-codigo`) and **signo** (`celda-signo`: `+`/`-`/`=`). 157 códigos. The `=`
totals delimit the sections used for grouping: `538` TOTAL DÉBITOS, `537` TOTAL
CRÉDITOS, `595` SUBTOTAL IMPUESTO DETERMINADO, `547` Total Determinado, `91` TOTAL A
PAGAR. Groups: `debitos` / `creditos` / `retenciones` / `otros` / `totales`. An
unobserved código → `otros` (surfaced, never hidden — anti-allowlist).

## Fase 2 — presented form via GWT-RPC (`rfiInternet`) — DEFERRED

The filed F29's full código grid (the real balance, incl. computed totals) is NOT in
the SDI JSON. It lives behind `rfiInternet` **GWT-RPC** (`text/x-gwt-rpc`,
pipe-delimited body, `//OK[...]` response). Mapped in the spike, to build later:

- **Warm-up flow:** Mi SII → `www.sii.cl/…1042-3266` → `sifmConsultaInternet` (GWT
  consulta app) → click → `rfiInternet/consulta/index.html` (GWT form viewer). A cold
  cookies-only call bounces to the login wall — the apps must be warmed via this nav
  chain + the `SdiAATokenService.getSdiAAInfoToken` handshake.
- **Key call:** `formularioFacade` op `findDeclaraciones(período)` → response embeds a
  clean `<FormularioRfi><Formulario><ListaCodigos><Codigos tipo="…"><Codigo>/<Valor>`
  XML. `tipo`: `M` = monto (tax line), `C` = text/PII (01 nombre, 06 dirección, 08
  comuna), `N`/`P`/`F` = other. `formularioFacade` op `cargarHtml` → the HTML render.
- **Brittleness:** build-specific strong-name hashes (e.g. the FormularioFacade
  strongName + Java type CRCs) change on every SII redeploy; two GWT apps redeploy
  independently. Hence Fase 2 must drive a real browser (harvest the live serialization
  context / intercept responses), not hand-construct GWT-RPC. Gated on a headless PoC.

## Quirks

- `formId` (estado) vs `formCodigo` (propuesta) — same internal id `"2"`.
- `resultadoCalculoPP29.traza` embeds the RUT — never surface it.
- `data:null` propuesta with no `metaData.errors` = "sin propuesta", not an error.
- No reCAPTCHA (the `rb_bf05726fov` POSTs are Dynatrace RUM beacons).
- The GWT-RPC **write** path (`formularioFacade` "Enviar Declaración") stays out of
  scope — needs its own legal ADR (ROADMAP "Write surfaces").
