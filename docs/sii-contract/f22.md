# Wire contract — F22 (Declaración Anual de Renta, estado)

Ported from the proven Python `sii-cli` (`portal/f22.py`) — first-hand observation,
no third-party library (ADR-004). Endpoints/payloads/fields cited from **spike #67**
(live-captured **2026-06-27**, prod, own session, off-season). All values below are
**synthetic / redacted** — folios, RUTs, montos, and ANY header código value are
PII/taxpayer data and MUST NOT land here. The **status** facades (`buscaDeclVgte` /
`f22Compacto`) are now **live-validated from the TS port** (#27, 2026-06-29, own account, AT
2023–2026); the **observaciones** facade (`situacionObservacion`) was captured AND
**live-validated** first-hand from TS on **2026-06-29** (spike #26) — a real session
returned observación `B102` with correct accents — see that section.

The **`formulario`** view (#27) was designed from a **first-hand headed-Playwright capture of the
SII "formulario completo" itself** (2026-06-29, own session, AT 2023–2026): the SPA renders
the full form from **`codigosFormato`** (the form skeleton) + **`f22Compacto`** (the values) +
`f22Compacto/pdf64` (the PDF). KEY FINDING: the sibling `f22Completo` endpoint returns
`f22Compacto`'s códigos PLUS internal control códigos (`3056/8891/8865…`) and extra PII
(`9306/9920`) — **noisier, not richer** — so `formulario` reads **`f22Compacto`** (same source as
the compact view) and only ADDS the contador grouping (see "Grouping"). `codigosFormato`
gives the authoritative header/PII set (`codigosCabecera`).

Surface: `@albertomarturelo/sii-core` `portal/f22.ts` (`fetchF22Declaraciones`, `fetchF22Grid`,
`groupCodigos`, `fetchF22Observaciones`, `fetchF22Historial`), driven by the
`f22Status` (`full?`) / `f22Overview` / `f22Observaciones` / `f22Historial` tasks under
`withSession`.
The F22-status portal is
the Angular SPA at `https://www4.sii.cl/consultaestadof22ui/` — a **different app** from
RCV (`consdcvinternetui`), same SDI envelope. No captcha (spike #67: zero recaptcha
fields, not even RCV's `t-o-k-e-n-web` sentinel).

**SESSION-KEYED (ADR-005).** F22 authorizes by the **session principal**, not the body
RUT. Confirmed live 2026-06-27: from a persona session, `--rut <empresa>` returned a
**clean negative** (`decls: null`), NOT the empresa's F22 — the body RUT does not
redirect it. In our model (no `use` verb) the empresa's F22 is reached by **logout→login
as the empresa**. The task takes NO `--rut`.

## Endpoints

Base: `https://www4.sii.cl/consultaestadof22ui/services/data/facadeService/`
(host from config `HOSTS.portalApi`). Envelope:
`{ metaData:{ namespace, conversationId, transactionId }, data:{…} }` (no `page`).
`f22 status` composes TWO under ONE session, paced (ADR-004); the 2nd POST is skipped
when no folio resolves.

| op | endpoint (`POST`) | namespace suffix |
| --- | --- | --- |
| declaraciones + estado | `buscaDeclVgte` | `…consultaestadof22.…FacadeService/buscaDeclVgte` |
| código grid (status + `formulario`) | `f22Compacto` | `…consultaestadof22.…FacadeService/f22Compacto` |
| form skeleton (sections/labels) | `codigosFormato` | `…consultaestadof22.…FacadeService/codigosFormato` |
| observaciones (inconsistencias) | `situacionObservacion` | `…consultaestadof22.…FacadeService/situacionObservacion` |
| historial / eventos (por folio) | `buscaEventos` | `…consultaestadof22.…FacadeService/buscaEventos` |

`codigosFormato` takes no `data` (returns the generic form template) → `data:{codigosCabecera[12],
codigosDinamicos[49], codigosPie[11]}`. `codigosCabecera` = the authoritative header/identity
códigos; we drop those (+ bank). The sibling `f22Completo` is NOT used (noisier, see above).

Namespace prefix: `cl.sii.sdi.lob.renta.consultaestadof22.data.api.interfaces.`.
The namespace was a best-guess (spike #67 captured URLs, not the exact string) but **SII
accepted it live 2026-06-27**; the URL path is authoritative for routing. Headers:
`Content-Type: application/json`, `Origin: https://www4.sii.cl`,
`Referer: https://www4.sii.cl/consultaestadof22ui/` (own Referer), `Accept: …`.

### `buscaDeclVgte` — request `data`

`{ "periodo": "2026", "rut": "<digits>", "dv": "<dv>" }` — `periodo` is the **año
tributario** (numeric YYYY); RUT split digits + DV.

Response `data`: `{ decls:[…], glosas:[…] }`.
- `decls[]` → curated `declaraciones[]` (`folio`, `vigente`=`vgte`, `estado` via
  `codConc`→`glosas`, `fecha`=`fecIng`, `tipoImpugnado`). The decl ALSO carries
  identity/bank fields (`nombres`/`calle`/`comuna`/`cta`/`bco`) — **dropped entirely**
  (see PII note below).
- `glosas[]`: `codConclusion` → `descripcion` (the human estado label).
- Empty/`null` `decls` = legitimate "sin declaración" (`tieneDeclaracion:false`), NOT an error.

### `f22Compacto` — request `data` (backs both `status` and `formulario`)

`{ "rut": "<digits>", "dv": "<dv>", "periodo": "2026", "folio": "<n>" }`. When `--folio`
is omitted the **vigente** declaración is auto-selected.

Response `data` is a flat LIST of `{ codigo, valor, glosa }` (glosas ship INLINE). Identity/
bank PII códigos → dropped (see PII note); the rest are the tax grid. `valor` coerced to
number, **sign preserved** (e.g. `305` resultado may be negative). This IS the form the SII
renders (and its PDF, `f22Compacto/pdf64`). Live row counts (own account, 2026-06-29): AT
2025 → 36 códigos, AT 2023 → 47, AT 2024 → 45 (honorarios years carry more lines).

#### Grouping — `formulario` organizes the grid into the lines a contador reads (`grupos`)

`groupCodigos` takes the PII-dropped grid and organizes it into **ingresos / deducciones /
retenciones·PPM·créditos / cálculo / resultado**, with an **`otros`** bucket for any non-PII
código not yet mapped — **VISIBLE, never hidden**, so the readback stays COMPLETE. The union of
the SIX equals the flat `F22Estado.codigos`. (`calculo` was split out of `resultado` in the #28
review: the intermediate IGC/IUSC steps — `157` según tabla, `158` subtotal, `304` débito
fiscal — are calc steps, not final outcomes; a "SUB TOTAL" is not a result.) This is a DENYLIST
posture: drop only identity/bank PII
(small, bounded, authoritative from `codigosFormato.codigosCabecera`), surface ALL tax content.
An allowlist was tried (#27) and REJECTED — it hid real honorarios/retenciones/deducciones
códigos for AT 2023–2024 (those weren't in the hand map), exactly the data a contador needs.

### `situacionObservacion` — request `data` (observaciones / inconsistencias)

`{ "periodo": 2026, "rut": "<digits>", "dv": "<dv>", "folio": <n> }` — **observed live
2026-06-29** (spike #26, own session). NOTE: `periodo` and `folio` go out as **NUMBERS**
here (the status facades send them as strings; SII accepts both).

Response `data` is a flat LIST of `{ codigo, descripcion, url }` — the observaciones that
flag the declaración:

- `codigo` — observación code, e.g. `B102` (ganancia de capital art. 107 LIR), `G37`
  (retiros/dividendos IGC). Curated.
- `descripcion` — the human glosa, verbatim.
- `url` — the SII ayuda PDF for correcting it
  (`http://www.sii.cl/ayudas/renta/ayuda_corregir_obs/<año>/<codigo>.pdf`).
- Empty `data` = **sin observaciones** (NOT an error).

**NOT PII** — unlike the f22Compacto grid, these rows carry only the observación code +
glosa + a public ayuda URL, so EVERY field is curated and there is **no header-código
exclusion and no `raw`**. The folio is resolved from `buscaDeclVgte` (vigente) or `--folio`.

Sibling endpoint `buscaObservacion` (per carta — needs `referencia`/`codigo`/`idCarta`
from `buscaEventos`) carries the per-notification detail; `situacionObservacion` gives the
vigente observaciones from the folio alone. The event TIMELINE is `buscaEventos` (below).

### `buscaEventos` — request `data` (historial / eventos)

`{ "periodo": "2025", "rut": "<digits>", "dv": "<dv>", "folio": "<n>" }` — **observed live
2026-06-29** (spike #28, own session). All go out as **STRINGS** (like `buscaDeclVgte`,
unlike `situacionObservacion`'s numbers). The **`folio` is REQUIRED**: posting
`{periodo,rut,dv}` without it returned `respCod: 2` + `errorMsg:
"RESTEASY001130: Error status 500 Internal Server Error returned"` (a framework error, not a
business one — `metaData` was null).

**Full SPA shape (headed capture 2026-06-29):** the real `consultaestadof22ui` app sends SIX
fields — `{periodo, rut, dv, folio, impugnado, evigCodigo}` — where `impugnado` =
`decls[].tipoImpugnado` and `evigCodigo` = `decls[].evigCodigo` for that folio. We send the
**`{periodo,rut,dv,folio}` subset**: live-verified it returns the SAME `data` the SPA gets for
the readable (vigente) folio, and the broken folio fails identically with or without the extra
fields — so the subset is sufficient and we don't thread decl context for zero gain (the full
shape is recorded here for fidelity, ADR-004).

Response `data` is a flat LIST of events for that folio (the SPA renders the "sección
Historial"). Observed fields per event:

- `codEvento` — the event-type code (e.g. `48` declaración recibida, `2` devolución
  autorizada, `10` depósito de Tesorería). Curated as `codigo`. NOTE the row ALSO carries an
  unrelated `codigo` (a long internal id) — the curated alias puts `codEvento` FIRST.
- `nombre` — the human event glosa, **verbatim** (e.g. "Su devolución solicitada por $… fue
  autorizada."). The **monto rides inside this text** — that is tax content (same class as the
  formulario montos), NOT an identity/bank field. Curated as `glosa`.
- `fechaEvento` — `DD/MM/YYYY`, verbatim. Curated as `fecha`. Wire order is oldest-first; the
  task sorts most-recent-first.
- `tipoEvento` → `tipo`; `codCarta` (notification number, leading zeros kept); `idCarta`
  (the per-carta key for `buscaObservacion` — often `null`, but a real value appears on carta
  events, e.g. `"13"` on an AT 2026 evento `505`); `referencia` (space-padded → trimmed);
  `fechaCitacion` / `unidadSii` (blank → null when absent).
- Empty / non-array `data` = **sin eventos** (NOT an error).

**NOT PII** — every field is a tax-process datum (event code/glosa/dates + carta refs), so
the row is fully curated, **no header-código exclusion and no `raw`** (same posture as
observaciones). The per-carta `buscaObservacion` drill (uses an event's non-null `idCarta`,
e.g. `"13"` above) is out of scope for `historial`; **`buscaEventos` IS the historial** — the
event timeline.

**Reach (the historial surface).** Default reads EVERY folio of the año (rectificatorias
included — buscaEventos is per-folio, so the task fans out over `decls[].folio`, paced via
`Clock.sleep`); `--folio` scopes to one. Events are aggregated across folios and sorted
most-recent-first. Error envelope: the top-level `errorMsg` (like `situacionObservacion`),
surfaced verbatim, never retried.

**Per-folio resilience.** Because the default reads several folios, ONE folio's `errorMsg`
must not bury the others. The task captures each failed folio's message **verbatim** in a
`foliosConError: [{folio, error}]` list and still returns the events from the folios that
succeeded — surfaced, never hidden, never retried (ADR-004). Only a session-level failure
(`NotAuthenticated`/`SessionExpired`, which is NOT an `F22Error`) aborts the whole read.

**Live-validated 2026-06-29** (own session, end-to-end through the CLI): AT 2025 → 3 eventos
(declaración recibida → devolución autorizada → depósito de Tesorería), AT 2024 → 5, AT 2023
→ 8, all most-recent-first with the montos inside each glosa and correct accents. **AT 2026
finding (the resilience case):** the año had TWO folios — the **vigente** (`vgte:"1"`, folio
…5546) returned its 2 rectificatoria eventos fine, while the **superseded** one (`vgte:"0"`,
folio …9396) returned a SERVER-SIDE parse error `errorMsg: "For input string: \"    <n>\""` (the
real digits redacted; the space-padding is the structural cause)
(SII's own Java `Integer.parse` choking on a space-padded internal value — our request is
well-formed; the vigente folio + 2023–2025 all succeed). The historial returned the vigente
folio's events **plus** the bad folio under `foliosConError` (CLI: a `⚠ folio … : <msg>`
line), exit 0. Also confirms **`vgte` arrives as `"1"`/`"0"`** here (earlier years sent
`"S"`/`"N"`; `coerceVigente` handles both — the contract's "unconfirmed form" is now resolved).

**Confirmed SII-side via headed capture (2026-06-29):** SII's OWN `consultaestadof22ui` app,
when the operator opened that folio's detail, fired the SAME `buscaEventos(…9396)` and got the
SAME `respCod 2 / "For input string: \"    <n>\""` — there is NO alternative endpoint for
a superseded folio's historial. So SII's website cannot render it either; our per-folio
resilience faithfully mirrors that (show the readable folios, flag the one SII can't serve).

**Live-validated 2026-06-29** via `sii f22 observaciones 2026` (own session): returned the
vigente observación `B102` (one row) with the descripción in correct accents — confirming
`situacionObservacion` serves the CURRENT outstanding observaciones (the per-carta history,
which also listed a since-resolved `G37`, lives in `buscaObservacion`).

## Error envelope

F22 signals a business error via **`metaData.errors`** — a LIST of `{ id, descripcion }`
(NOT RCV's `respEstado.codRespuesta`). Non-empty ⇒ `F22Error` with `descripcion`
**verbatim** (ADR-004), never retried. Empty/absent ⇒ no error.

`situacionObservacion` ALSO carries a top-level `errorMsg` (string) + `respCod` alongside
`metaData.errors` (observed 2026-06-29, both `null` on success). The facade surfaces a
non-empty `errorMsg` verbatim as `F22Error` too.

## PII — only identity/bank códigos are DROPPED; all tax content is surfaced

The f22Compacto grid mixes tax códigos with **identity / bank** códigos. ONLY the latter are
excluded (`HEADER_CODIGOS`), authoritatively grounded in `codigosFormato.codigosCabecera`
(the form's header section = `1/2/3/5/6/7/8/9/13/14/55/903`) plus the date/región/moneda
metadata (`15/53/315/8811`) and the bank/address códigos that sit in the body: **`301/306/780`**
(banco / cuenta / tipo — caught leaking 2026-06-27), **`9306`** (código de banco), **`9920`**
(dirección origen), and **`8809`** (the RUT again as a bare integer, body+DV — caught leaking
via test **MF3**, 2026-06-29). The decls[] identity/bank fields are dropped too.

Unlike RCV, **F22 exposes NO `raw`** — but the curated grid surfaces **every TAX código**
(ingresos, honorarios, retenciones, PPM, deducciones, créditos, resultado). Only identity/bank
PII is withheld (it reaches the LLM via MCP — ADR-006). An **allowlist** was tried for `formulario`
(#27) and REJECTED: it dropped real honorarios/retenciones/deducciones códigos for AT 2023–2024
(absent from the hand map) — the exact data a contador needs. So the posture is a **denylist of
identity/bank only**; everything else (incl. unmapped tax códigos) is shown, unmapped ones via
the `otros` group. The PII set is bounded + authoritative (the form cabecera), so it stays
comprehensive without hiding tax content.

## Reach (ADR-005, spike #15 answer for F22)

**F22 is session-keyed — `operate`/`--rut` does NOT reach a represented empresa's F22**
(confirmed live: clean negative under the persona session). The empresa's F22 needs the
empresa's own session (logout→login). The task ignores the operate pointer and reads the
session principal.
